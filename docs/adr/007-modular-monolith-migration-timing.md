# ADR-007: モジュラーモノリス移行のタイミング

## ステータス

提案中

## コンテキスト

本リポジトリの中心テーマはモジュラーモノリス（以下 MM）の練習であり、`TODO.md` では Phase 1（基本的な EC 機能）を完了した後に Phase 2（モジュール分離）に着手する方針を置いていた。

現状:

- Phase 1 はおおむね完了（会員、商品 CRUD、カート、注文、クーポン）
- Phase 2 は未着手
- Phase 3（レビュー、在庫管理）は未実装

`TODO.md` には「最低限の EC サイトを作る。どうやってモジュラモノリスにするかの検討はそこまでできてからやる」とあるが、「最低限」がどこまでかは未定義のままであり、移行に踏み切る具体的な条件が決まっていない。

### 移行タイミングを判断する枠組み

「いつ移行するか」の難しさは本質的に「やり直しコストが高い時、後悔したくない」という不可逆性に由来する。本リポジトリでは以下の前提で**可逆性を確保できる**ため、判断軸は「最も早く・多く学べる時期はいつか」に単純化される:

- monolith は別ブランチで active に維持し続ける（凍結しない）
- 任意時点の monolith から派生して再 MM 化を試せる
- ある MM パターンの境界が破綻しても、monolith に戻って別パターンで再着手できる

具体的なブランチ命名・PR 運用・反映方針は別 ADR で詳細化するが、「monolith を生かし続け、いつでも再分岐できる」という大方針は本 ADR の前提として確定させる。

### 検討した移行タイミングの候補

1. **今すぐ着手**（現状 = Phase 1 完了時点）
2. **在庫管理だけ追加してから着手**（在庫は注文との結合が強くモジュール境界の典型題材）
3. **Phase 3 全機能（レビュー＋在庫）完了後に着手**

## 決定

**今すぐ MM 移行に着手する。**

可逆性が確保されているため「走りながら直す」が成立する。Phase 3 の機能追加は MM 化後にも継続でき、かつ「既存の MM 構造に新機能を後から組み込む」過程そのものが MM 学習の中核である。移行を遅らせる積極的な理由はない。

## 理由

### 1. 移行コストが可逆である（前提から導かれる結論）

ブランチ運用により monolith は active 維持されるため、MM 化の方針が誤っていた場合でも monolith ブランチから再分岐して別パターンで着手できる。「より複雑な monolith からの再 MM 化」を将来試したくなった場合も、その時点の monolith から `git checkout -b mm-N` で派生できる。

つまり「今移行を始めること」で失う選択肢はほぼ無い。タイミング判断における最大のリスク要因（不可逆性）が解消されているため、「待つ」ことの相対価値が下がる。

### 2. 「既存 MM 設計に新機能を後から取り込む」体験が MM 学習の本丸

実務で MM を扱う際に最も頻出するのは「ゼロから境界を引く」場面ではなく「既存のモジュール構造に新機能をどう組み込むか」の判断である。今移行に着手し、その後に在庫管理などを追加していく流れは、この実務シナリオを直接練習できる。

### 3. 在庫追加時の設計判断を「実装で」検証できる

注文 → 在庫減算のようなモジュール跨ぎの状態変更を、直接呼び出し／インターフェース／イベント駆動のどれで吸収するか。MM 化済みの状態で在庫を追加することで、各方式の実装コストや破綻リスクを実物で確認できる。MM 化前に在庫を入れてしまうと、この判断機会を失う。

### 4. 境界の妥当性は機能追加で検証されるべき

MM 化前にどれだけ Phase 3 の機能を増やしても「境界が正しく引けたか」は実装してみるまで分からない。逆に、現時点の少ない機能で引いた境界が後の機能追加で破綻するなら、その「破綻」自体が学習材料になる。境界を引き直す経験も、別パターンで再着手する経験も、本リポジトリの練習対象に含まれる。

### 却下した代替案

- **在庫管理だけ追加してから着手**: 在庫は確かに MM の良い題材だが、それを MM 化前に入れてしまうと「既存 MM に在庫をどう組み込むか」という最大の学習機会を失う
- **Phase 3 全機能完了後に着手**: レビュー機能は MM の論点を強く生まないため、待つコストに見合う学習価値が薄い。また monolith の機能数が増えるほど初回 MM 化のコストも線形に増える

## 影響

- `TODO.md` の Phase 2 着手条件（暗黙的に「Phase 1 完了 + α」とされていた部分）は本 ADR で「今着手」に確定する
- 「monolith を active 維持し、任意時点から再 MM 化可能とする」運用方針は本 ADR で確定とし、後続 ADR の前提とする
- 移行**方法**（モジュール分割の粒度、インターフェース定義方式、イベント基盤の有無）は別 ADR で決定する
- 管理運用の**詳細**（ブランチ命名、PR 運用、反映方針、worktree や apps/ 構成の採否）も別 ADR で決定する
- Phase 3（レビュー、在庫管理）は MM 化後に MM 構造上で実装する前提となる
