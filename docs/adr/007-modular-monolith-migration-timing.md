# ADR-007: モジュラモノリス移行のタイミング

## ステータス

採用

## コンテキスト

このリポジトリはモジュラモノリスの学習・実験が主目的であり、`TODO.md` では Phase 1（EC 基本機能の実装）完了後に Phase 2（モジュール分割・イベント基盤）に進む計画になっている。

Phase 1 は完了し、現状は以下の規模:
- ドメイン 5 つ: 認証 / 商品 / カート / 注文 / クーポン
- モデル 11 個、コントローラ 8 個（`app/controllers/api/v1/`）
- `Order` が `User` / `Product` / `Coupon` を跨いで参照しており、境界設計の難所が既に存在する
- モジュール化の試みは未着手（`backend/app/` 配下はフラットな Rails 標準構造）

ここで判断が必要になったのは「今すぐ Phase 2 に入るか、Phase 3（レビュー・評価、在庫管理）で複雑さを増してから入るか」。
「もう少し複雑さがあった方が境界設計の題材として良いのでは」という懸念と「もう十分複雑なのでは」という感覚の両方がある状態。

## 検討した案

### 案A: 今すぐ Phase 2 に入る

Phase 3 の機能追加はせず、現状の 5 ドメインで分割を開始する。

- メリット:
  - 5 ドメイン・11 モデルは境界設計を学ぶ題材として既に十分（特に `Order` の跨ぎ参照）
  - コード量が少ない段階で分割すれば移行コストが低く、失敗しても戻しやすい
  - Phase 3 の機能追加を「分割後に新モジュールを足す練習」にできる（実運用に近い流れ）
- デメリット:
  - ドメインが少なく、分割の恩恵（独立デプロイなど）は体感しづらい。ただし学習目的では問題にならない

### 案B: Phase 3（レビュー・評価、在庫管理）を入れてから移行

機能を追加して複雑さを増してから分割する。

- メリット:
  - モジュール数が増え、境界の引き方の選択肢が広がる
  - 分割の必要性をより実感できる
- デメリット:
  - フラット構造のまま機能を積むと、後の分割が「全ファイル触る大リファクタ」になる
  - 学びが苦行化し、モチベーション維持が難しい
  - Phase 3 機能自体はフラット構造でも実装可能で、分割後に足した方が学習効果が高い

### 案C: 在庫管理だけ先に追加してから移行

Phase 3 の一部（在庫）のみ先行し、複雑さを少し増してから分割する。

- メリット:
  - 在庫は商品と注文の両方に絡むため、境界設計の題材として有用
- デメリット:
  - 案Bと同じく、分割コストが上がるデメリットは残る
  - 在庫は分割後に入れても題材としての価値は変わらない

## 決定

**案Aを採用。今すぐ Phase 2（モジュール分割）に入る。**

## 採用理由

- 現状の 5 ドメイン・11 モデルは境界設計を学ぶのに十分な複雑さがある。特に `Order` が `User` / `Product` / `Coupon` を跨いで参照している構造は、モジュール間のインターフェース設計の典型的な題材になる
- コード量が少ない今の方が移行コストが低く、失敗時のロールバックも容易
- Phase 3 を先に入れるとフラット構造のまま機能が増え、後の分割が大規模リファクタ化する。学習が苦行になるとモチベーションを損なう
- Phase 3（レビュー・在庫）は分割完了後に「新しいモジュールを追加する練習」として実装する方が、実運用（既存モジュラモノリスに機能を足す）に近い流れになる
- 「分割してみて戻す」も学習成果として受容できる状態（個人学習リポジトリ）なので、早めに着手してフィードバックを得る方が総合的な学びが大きい

## 影響

- Phase 2 の作業（モジュール境界の決定、技術選択、分割実装）に着手する
- 境界の引き方・技術選択（Packwerk / Rails Engine / 名前空間など）は別 ADR で決定する
- Phase 3（レビュー・評価、在庫管理）は Phase 2 完了後に着手する
- イベント基盤の導入は Phase 2 完了後に別途検討する
