# ADR-007: モジュラモノリス移行の開始タイミング

## ステータス

採用

## コンテキスト

このリポジトリの主目的はモジュラモノリスアーキテクチャの練習である。
これまで移行を先送りしていた理由は「もっと複雑にしてから移行したほうが練習になる」と考えていたため。

最近、**monolith ブランチと modular-monolith ブランチを分けて運用する**案を思いついた。
monolith 側に機能追加 → modular 側に反映、という流れを取れば、
「複雑化した状態での移行」も後から繰り返し練習できる。
従って「複雑化を待つ」理由は弱まり、開始タイミングを再評価する必要が出た。

現状の規模・整備状況：

- モデル 11 / コントローラ 8 / リレーション 23 本
- ドメインは概ね 5 領域: User / Product / Cart / Order / Coupon
- packwerk・Rails Engines は未導入。namespace は API 層 (`Api::V1`) のみ
- ADR 6 件で設計判断の蓄積あり（`docs/adr/001`〜`006`）
- テスト・CI 整備済み（RSpec / vitest / GitHub Actions）
- CLAUDE.md に「モジュール間の依存は明示的に管理」「モジュール境界を越える操作はインターフェース経由で行う」と方針言語化済み

## 検討した案

### 案A: 今すぐ移行を開始する

- メリット:
  - 5 ドメイン・23 リレーションは「境界の引き方で悩める」最低ラインに達している
  - ブランチ運用案は早期分岐ほど初回反映の差分が小さくなり、運用が持続可能
  - 「今の規模での移行」「機能追加後の追従移行」を複数回練習できる
  - 設計方針が言語化済みで、意図を持った移行ができる
- デメリット:
  - これ以上複雑化してから始めるのと比べ、初回移行の難易度はやや低い

### 案B: あと数機能追加してから開始する

- メリット: 初回移行時の境界判断がより難しくなり、悩みごたえが増える
- デメリット:
  - 「悩みの種類」はあまり変わらず「悩みの量」が増えるだけ
  - 分岐が遅れるほどブランチ運用案の初回反映コストが大きくなり、運用が破綻しやすい
  - 練習回数が減る

### 案C: 移行はせず、monolith のまま機能追加を続ける

- メリット: 機能追加に集中できる
- デメリット: リポジトリの主目的（モジュラモノリス練習）から外れる

## 決定

**案A を採用。今のタイミングでモジュラモノリス移行を開始する。**

## 採用理由

- 練習に必要な最低限の複雑度は既に達成している
- ブランチ運用案そのものが「早く分岐すること」を要求する（遅らせるほど運用が成立しにくい）
- 複数回の移行練習が積めるほうが学習効率が高い
- 設計方針の言語化が済んでいる今が好機

## 影響

- 「ブランチ運用方針（main をどちら扱いにするか等）」と「移行の初手（packwerk / ディレクトリ分割 / Engine など）」は別 ADR で決定する
- 最初に切り出す対象ドメインは別途検討する（独立度の高い Coupon が候補）
- 既存の RSpec / vitest / CI はグリーンを維持する前提で進める
