# ADR（Architecture Decision Record）ガイドライン

## ADR とは

ADR は、プロダクトやシステムに関する1つの決定を記録・説明する短いドキュメント。決定の内容、その背景、トレードオフを簡潔にまとめる。

参考: [Martin Fowler - Architecture Decision Record (2026)](https://martinfowler.com/articles/architecture-decision-record.html)

## このドキュメントについて

このガイドラインは暫定的なものであり、運用しながら随時改善していく。

ただし、個別の ADR は決定の記録であり、一度「採用」になったら変更しない。内容を覆す場合は新しい ADR を作成し、元の ADR を「廃止」にする。

## 基本ルール

- **1つの決定につき1つの ADR** を書く
- **短く書く**（1〜2ページが目安）
- **逆ピラミッド型** で書く — 最も重要な情報を先に、詳細は後に
- **採用済みの ADR は変更しない** — 内容を覆す場合は新しい ADR で置き換え、元の ADR は「廃止」にする

## ファイル規則

- **配置場所**: `docs/adr/`
- **ファイル名**: `NNN-descriptive-name.md`（例: `004-payment-gateway-selection.md`）
  - 3桁の連番
  - 英語・小文字・ハイフン区切り
- **本文**: 日本語で書く

## ステータス

| ステータス | 意味 |
|-----------|------|
| 提案中 | レビュー待ち |
| 採用 | 承認済み、現在有効 |
| 廃止 | 新しい ADR によって置き換えられた（後継 ADR へのリンクを記載） |

### 決定を覆す場合

1. 新しい ADR を作成し、コンテキストになぜ前の決定を覆すのかを書く
2. 元の ADR のステータスのみ `廃止（→ ADR-NNN）` に変更する（本文は一切変更しない）

## テンプレート

以下をコピーして使用する。

```markdown
# ADR-NNN: タイトル

## ステータス

提案中

## コンテキスト

なぜこの決定が必要になったのか。背景、課題、制約を説明する。

## 決定

何を決めたか。

## 理由

なぜそう決めたか。検討した代替案とそのトレードオフを含める。

## 影響

（任意）決定による影響や、見直しを検討すべき条件があれば記載する。
```
