## 1. 境界の引き方（モジュール数・名前・所属モデル）

**ステータス**: 未決定

決めること:
- モジュールを何個に分けるか
- 各モジュールの名前
- `backend/app/models/` の 11 モデルをどのモジュールに割り当てるか

現状の候補:
- auth / catalog / shopping / promotion の 4 モジュール（plan ファイルで挙げた推奨案）

判断の難所:
- Cart と Orders を別モジュールにするか、同一購入フローとしてまとめるか
- Coupon は Product の付属か独立ドメインか（現状 `Coupon belongs_to :product`）
- ProductImage は Product 内部のサブ概念か別モジュールか

決定方法: **ADR-008 として記録する**

## 4. 技術選択

**ステータス**: 未決定

決めること:
- Packwerk / Rails Engine / 名前空間のみ / その他 のどれにするか

候補の特徴:
- Packwerk: 静的検査で境界違反を検知。gem 追加だけで始められる。学習題材として適切
- Rails Engine: 最も厳密な分離。routes/migration/Gemspec が engine ごとに必要で学習コスト高
- 名前空間のみ: 導入コスト最小だが違反を防ぐ仕組みなし

決定順序: 境界の引き方（§1）を先に決めてから判断

決定方法: **ADR-009 として記録する**

## 6. 移行順序

**ステータス**: 未決定

決めること:
- パイロット 1 モジュールから始めるか、一気に分けるか
- どのモジュールから着手するか

候補:
- 被参照側（catalog / products）から: 違反が少なく、Public API 設計の練習に集中できる
- 最難関（orders）から: 境界の引き直しが多発するリスクが高いが学びも大きい
- 一気に並行分割: 違反が大量発生しやすい

決定方法: design メモとして残す
